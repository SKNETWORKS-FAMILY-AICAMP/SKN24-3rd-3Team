
## RAG용 문서 데이터 수집 및 전처리

### 1. RAG용 문서 데이터 수집

본 프로젝트는 건설·산업안전 분야 질의응답의 신뢰성을 확보하기 위해 **KOSHA GUIDE PDF 문서**를 RAG의 기반 문서로 활용하였다. KOSHA GUIDE는 작업 공정별 안전기준, 사고 예방 수칙, 작업 환경별 조치사항 등 현장에서 참고할 수 있는 공식 지침을 포함하고 있어, 사용자의 질의를 신뢰도 높은 기준 문서와 연결하는 데 적합하다고 판단하였다.

또한 본 프로젝트는 단순히 답변을 생성하는 것에 그치지 않고, **답변의 근거가 되는 문서를 함께 제시하는 설명 가능한 질의응답 시스템**을 목표로 하였다. 이에 따라 일반 생성형 응답보다 공식 기준에 기반한 답변이 가능하고, 출처를 명확히 제시할 수 있는 KOSHA GUIDE 문서를 원본 데이터로 선정하였다.

수집 데이터는 PDF 형식의 비정형 문서이며, 문서 내에는 본문 텍스트뿐 아니라 표, 그림 캡션, 문서 번호, 페이지 번호, 목차, 부록, 개정이력 등 다양한 부가 요소가 함께 포함되어 있다. 따라서 원본 데이터를 그대로 검색에 활용하기보다는, 문서의 구조와 의미를 유지하면서 검색에 적합한 형태로 정제하는 전처리 과정이 필요하였다.

**출처 링크**

- [KOSHA 안전규정집](https://portal.kosha.or.kr/archive/resources/tech-support/search/const?page=1&rowsPerPage=10)

### 2. 원본 데이터의 특성 및 전처리 필요성

원본 KOSHA GUIDE PDF는 안전 작업 기준과 예방 수칙을 포함한 공식 문서이지만, 검색 및 RAG 파이프라인에 바로 적용하기에는 몇 가지 한계가 있었다.

첫째, 원본 PDF에는 **헤더, 문서 번호, 페이지 번호, 목차, 개정이력, 부록 정보**와 같이 본문 의미와 직접적인 관련이 없는 반복 텍스트가 포함되어 있었다. 이러한 요소들은 검색 시 질의와 무관한 텍스트를 상위에 노출시킬 수 있어 검색 정확도를 저하시킬 가능성이 있었다.   
<img width="635" height="891" alt="image" src="https://github.com/user-attachments/assets/a0e2b264-b7d5-4fa3-a61e-5f67b1b83b35" />

둘째, 문서 내에는 본문과 표가 혼재되어 있었다. KOSHA GUIDE는 작업 기준, 수치 기준, 비교 항목 등이 표 형태로 제시되는 경우가 많으므로, 본문 텍스트만 추출할 경우 일부 핵심 정보를 놓칠 수 있다. 따라서 PDF 내 표 데이터 역시 함께 추출하여 활용할 필요가 있었다.   
<img width="635" height="898" alt="image (1)" src="https://github.com/user-attachments/assets/a8657650-a658-41ac-bf77-b909ee5eb895" />

셋째, KOSHA GUIDE는 `7 → 7.1 → 7.1.1`과 같은 **숫자 기반 계층형 헤딩 구조**를 가지고 있다. 이는 문서가 단순한 텍스트 집합이 아니라 상위 개념과 하위 세부 항목이 구조적으로 연결되어 있음을 의미한다. 따라서 일정 길이 단위로 단순 분할하는 방식보다는, 문서 구조를 반영하여 문맥을 유지할 수 있는 청킹 방식이 필요하였다.   

<img width="633" height="893" alt="image (2)" src="https://github.com/user-attachments/assets/e74a3c09-e96e-4e97-926d-0a6561417e4f" />

넷째, 본 프로젝트는 단순 키워드 검색이 아니라 사용자의 자연어 질문에 대해 근거 기반 답변을 생성하는 RAG 시스템을 목표로 하였다. 이를 위해서는 검색 단계에서 세부 내용을 정교하게 찾을 수 있어야 하고, 답변 생성 단계에서는 관련 조항의 상위 문맥을 함께 활용할 수 있어야 한다. 따라서 검색용 청크와 답변 생성용 청크를 분리하는 구조가 필요하였다.

이러한 이유로 본 프로젝트에서는 **노이즈 제거**, **문서 구조 기반 청킹**, **메타데이터 태깅**, **JSON 데이터셋 구축**의 과정을 포함하는 전처리를 수행하였다.

### 3. 전처리 과정

### 1) PDF 텍스트 및 표 추출

원본 PDF 문서의 본문 정보와 표 정보를 함께 활용하기 위해 `pdfplumber`를 사용하여 문서 내 텍스트와 표 데이터를 추출하였다. 이를 통해 본문에 포함된 서술형 정보뿐 아니라, 표에 제시된 수치 기준이나 비교 항목도 검색 가능한 텍스트 형태로 변환하였다.

### 2) 불필요한 노이즈 제거

검색 성능을 높이고 문서 구조를 최대한 유지하기 위해 줄 단위 및 페이지 단위 노이즈 제거를 수행하였다.

줄 단위 제거 대상은 다음과 같다.

- `KOSHA GUIDE` : 문서 상단 헤더
- `D-C-7-2026` : 문서 번호
- `1 -` : 페이지 번호
- `<그림 1>`, `그림 1.` : 그림 캡션
- `목 차` : 목차 제목
- 개정이력, 부록 관련 텍스트

또한 다음과 같은 페이지는 문서 본문보다 부가 정보 성격이 강하다고 판단하여 페이지 전체를 제외하였다.

- 부록 페이지
- 개정이력 페이지
- 제안개요 페이지
- 심의개요 페이지

이 과정에서 `is_noise()` 함수는 각 줄이 제거 대상 노이즈인지 판별하는 역할을 수행하였고, `should_skip_page()` 함수는 해당 페이지 전체를 제외할지 여부를 판별하는 역할을 수행하였다.

이러한 노이즈 제거 과정을 통해 검색과 직접적으로 관련된 본문 중심의 데이터만 남기고, 반복적이고 비의미적인 텍스트는 제거할 수 있었다.

### 3) 헤딩 인식 및 구조화

문서의 상위 문맥을 유지하기 위해 숫자로 시작하는 줄을 헤딩으로 인식하였다. 이를 위해 `heading_pattern` 정규식을 사용하여 숫자 기반의 제목 구조를 판별하였으며, `parse_heading()` 함수를 통해 헤딩일 경우 `(depth, 번호, 제목)`을 반환하고, 일반 문장일 경우 `None`을 반환하도록 구성하였다.

헤딩의 depth는 번호의 점(`.`) 개수를 기준으로 결정하였다.

- `7` → depth 1
- `7.1` → depth 2
- `7.1.1` → depth 3

이 과정을 통해 문서가 갖고 있는 상위-하위 구조를 보존한 상태로 이후 청킹 과정을 수행할 수 있도록 하였다.

### 4) 부모 청크 생성

숫자 기반 헤딩 구조를 바탕으로 문서의 상위 문맥을 유지하는 **부모 청크(Parent Chunk)** 를 생성하였다. 부모 청크는 하나의 상위 주제 또는 절 단위를 대표하며, 관련 세부 항목들을 포괄하는 문맥 단위로 구성하였다.

부모 청크는 이후 RAG 답변 생성 단계에서 활용되며, 검색 결과로 선택된 세부 청크가 어떤 문맥 안에 위치하는지 확인할 수 있도록 하는 역할을 수행한다.

### 5) 자식 청크 생성

부모 청크 내부에서는 `(1)`, `(가)`, `①`과 같은 항목 표현을 기준으로 내용을 세분화하여 **자식 청크(Child Chunk)** 를 생성하였다. 자식 청크는 보다 작은 의미 단위로 분리된 검색용 청크이며, 사용자의 질문과 직접적으로 관련된 세부 내용을 탐색하는 데 활용된다.

즉, 부모 청크가 상위 문맥 유지에 초점을 맞춘다면, 자식 청크는 세부 검색 정확도 향상에 초점을 맞춘 구조라고 볼 수 있다.

### 6) 청크 태깅 및 카테고리 분류

청크 생성 이후 검색 및 분석 편의성을 높이기 위해 추가적인 태깅을 수행하였다.

- **Task**와 **Space**는 사전 정의한 키워드 규칙을 기반으로 자동 분류하였다.
- **Category**와 **Keyword**는 API를 활용하여 생성 및 태깅하였다.
- 자식 청크에는 부모 청크의 태그를 복사하여 태깅의 일관성을 유지하였다.

여기서 Task와 Space는 미리 정의한 작업 유형 및 공간 관련 키워드 목록을 기반으로 자동 할당하였고, Category와 Keyword는 문서 의미를 보다 유연하게 반영할 수 있도록 API를 활용해 생성하였다.

이러한 태깅 과정을 통해 각 청크에는 단순 본문 텍스트 외에도 작업 유형, 작업 공간, 주제 카테고리, 핵심 키워드 등의 메타데이터가 부여되었으며, 이는 이후 검색 고도화와 문서 해석에 활용할 수 있도록 하였다.

### 7) JSON 데이터셋 구축

최종적으로 전처리된 데이터는 **Parent-Child 구조의 JSON 데이터셋**으로 저장하였다. 각 청크에는 다음과 같은 메타데이터를 포함하였다.

- `chunk_id`
- `parent_id`
- `depth`
- 문서명
- 제목
- 본문 내용
- Task
- Space
- Category
- Keyword

이와 같은 구조를 통해 원본 PDF 문서를 검색과 답변 생성에 적합한 형태의 구조화 데이터로 변환하였다.

---

### 4. 전처리 전후 데이터 비교

원본 데이터는 비정형 PDF 형식으로 존재하며, 문서 내에는 헤더, 페이지 번호, 문서 번호, 목차, 부록, 개정이력 등 검색과 직접 관련이 없는 요소가 함께 포함되어 있었다. 또한 문서가 계층형 구조를 가지더라도, PDF 상태에서는 이러한 구조를 검색 시스템이 직접 활용하기 어려웠다.

반면 전처리 이후 데이터는 다음과 같은 차이를 갖는다.

- 원본 PDF의 반복 노이즈 제거
- 본문 및 표 데이터 추출
- 숫자 기반 헤딩 구조 유지
- Parent-Child 구조의 청크 생성
- Task, Space, Category, Keyword 태깅 부여
- JSON 형태의 구조화 데이터셋 구축

즉, 원본 데이터가 사람이 읽기 위한 비정형 문서였다면, 전처리 이후 데이터는 **검색과 답변 생성에 최적화된 구조화 데이터**로 변환되었다고 볼 수 있다.

또한 전처리 결과 예시로 Parent Chunk와 Child Chunk 샘플을 각각 제시하여, 최종 데이터가 어떤 형태로 저장되는지 확인할 수 있도록 하였다. Parent Chunk는 상위 문맥을 포함한 답변 생성용 데이터이고, Child Chunk는 세부 검색에 활용되는 검색용 데이터이다.

#### Parent Chunk 예시
```json
{
  "chunk_id": "f504de12fe52",
  "chunk_type": "parent",
  "source": "C-06-2015 타일(Tile)공사 안전보건작업 지침.pdf",
  "title": "타일(Tile)공사 안전보건작업 지침",
  "year": 2015,
  "status": "Active",
  "depth_1": "3 용어의 정의",
  "depth_2": "",
  "depth_3": "",
  "category": "작업 절차",
  "task": "조적미장",
  "space": "내부",
  "keyword": [
    "타일공사",
    "타일",
    "타일 붙이기"
  ],
  "content": "(1) 이 지침에서 사용하는 용어의 정의는 다음과 같다.\n(가) “타일공사(Tiling, Tile works)”란 각종 타일붙이기에 관한 공사를 말한다.\n(나) “타일(Tile)”이란 점토나 암석의 분말(또는 도토, 자토)을 성형 소성하여 만든 얇은 판형의 것을 말하고, 보통 타일은 도자기 타일을 말한다.\n(다) “타일의 종류”란 아래와 같이 분류한다.\n① 원재료의 질(質)에 의한 분류\n㉮ 도기질\n㉯ 자기질\n㉰ 반자기질 등\n② 제(조)법에 의한 분류\n㉮ 습식타일\n㉯ 건식타일\n㉰ 반건식타일\n③ 용도상에 의한 분류\n㉮ 외부용(벽체 보호상 흡수율이 적은 것)\n㉯ 내부용(벽, 바닥, 천정용)\n(라) “타일나누기”란 도면 또는 관리감독자의 지시에 따라 수준기, 레벨 및 다림추 등을 사용하여 기준선을 정하고 될 수 있는 대로 온장을 사용할 수 있도록 사전에 타일 부착 면에 나누기를 하는 것을 말한다.\n(마) “타일 붙이기”란 현장 배합 모르타르, 기성 배합 모르타르 및 접착제 등 붙임 재료를 사용하여 바탕 면에 타일을 붙이는 것을 말한다.\n(바) “붙임시간(Open Time)”이란 접착 모르타르나 접착제를 바탕면 또는 타일면에 발라 타일붙이기 하기에 적당한 상태가 되기까지의 최대 한계시간을 말한다.\n(사) 그 밖의 이 지침에서 사용하는 용어의 정의는 특별한 경우를 제외하고는 산업안전보건법, 같은 법 시행령, 같은 법 시행규칙, 안전보건규칙 및 관련 고시에서 정하는 바에 따른다."
}
```

| 포함 정보 | 헤더, 문서 번호, 페이지 번호, 목차, 본문, 표, 부록/개정이력 혼재 | 본문 및 표 추출, 노이즈 제거, 메타데이터 포함 |
| 문서 구조 | 사람이 읽기 위한 문서 구조 | Parent-Child 청크 구조 |
| 텍스트 상태 | 반복 노이즈와 비의미 텍스트 포함 | 검색에 필요한 핵심 본문 중심으로 정제 |
| 문맥 유지 방식 | PDF 원문 흐름에 의존 | 헤딩 기반 계층 구조와 depth 정보 유지 |
| 청크 단위 | 별도 청크 구조 없음 | Parent Chunk와 Child Chunk로 분리 |
| 메타데이터 | 별도 구조화 정보 없음 | chunk_id, parent_id, depth, 문서명, Task, Space, Category, Keyword 포함 |
| 활용 방식 | 직접 검색 및 활용이 어려움 | 검색 및 RAG 답변 생성에 활용 가능 |

### 5. 최종 데이터셋 구성 및 활용 방식

최종 데이터셋은 Parent Chunk와 Child Chunk의 이중 구조로 구성하였다.

- **Parent Chunk**: 상위 헤딩과 문맥을 유지한 답변 생성용 청크
- **Child Chunk**: 세부 항목 단위로 분리된 검색용 청크

검색 단계에서는 Child Chunk를 활용하여 사용자의 질의와 가장 관련성이 높은 세부 내용을 탐색하였다. 이후 답변 생성 단계에서는 검색된 Child Chunk가 속한 Parent Chunk를 참조하여, 상위 문맥이 유지된 상태에서 근거 기반 답변을 생성하도록 구성하였다.

이를 통해 단순 유사도 검색만 수행하는 방식보다, 문서의 세부 검색 정확도와 상위 문맥 반영을 동시에 확보할 수 있도록 하였다.
